# E-0008: M6.1 — Loop Legibility and UX Clarity

## Metadata

- Status: active

## Goal
Fix the UX clarity gaps discovered during E-0007's tier-2 comprehension probe so that the primary user can explain what the observe-propose-validate loop does within ~10 seconds, and the key affordances (feedback on a response, proposal form, suggestions panel) feel purposeful rather than overwhelming. This milestone gates M7 expansion of improvement classes.

## Rationale
- E-0007 closed with a failed comprehension gate. Three concrete blockers were identified by the primary user (F-20260301-006):
  1. Feedback button navigates to the top of settings, not a focused feedback view.
  2. "Feedback" heading appears twice in the feedback panel.
  3. Proposal form is overwhelming: all fields visible simultaneously, no purpose explanation, looks like an admin form.
- These are not conceptual failures — the loop infrastructure works. They are UX/legibility failures that prevent users from understanding what the loop does and trusting it.
- Fixing these before M7 scope expansion is the correct sequencing: "ship one clear thing" before scaling complexity.

## Definition of Done
1. **Duplicate heading fixed**: "Feedback" heading does not appear twice in the feedback panel (T-0048).
2. **Feedback navigation direct**: clicking feedback on a response lands the user directly at the feedback form, not the top of settings (T-0049).
3. **Proposal form simplified**: form shows a purpose description, Title + Rationale prominently, and hides advanced fields by default (T-0050).
4. **Comprehension gate re-passed (tier-2)**: after M6.1 ships, run a targeted re-probe. The primary user can explain what the loop does within ~10 seconds.

## Validation Plan

### Tier 1 (deterministic)
- Automated tests pass for modified UI components (settingsPanel.test.tsx and any new tests).
- Manual check: feedback button → feedback form visible immediately; "Feedback" heading appears once; proposal form shows purpose description and hides advanced fields by default.
- Evidence: QA checkpoint in `tickets/meta/qa/`.

### Tier 2 (micro-validation — comprehension re-gate)
- Audience: primary user / project sponsor.
- Run after M6.1 batch ships.
- Probe (single, focused): "You see a feedback button next to an AI answer. What do you think clicking it does? And once you're in the suggestions area, what do you think happens next?"
- Pass criterion: user can describe the loop in plain terms within ~10 seconds.
- Where results recorded: implementing ticket's Evidence section and PM checkpoint in `tickets/meta/feedback/`.
- Decision: pass → proceed to M7 improvement class expansion. Fail → one more focused UX iteration.

## Risks
- Scope creep: do not redesign the full settings panel. Fix the three identified issues only.
- Over-engineering T-0049: prefer scroll-to-section over modal/overlay unless scroll clearly doesn't solve it.

## Non-goals
- Redesigning the settings panel layout or navigation.
- Adding new improvement classes (M7 scope).
- Changing proposal artifact schema or acceptance/validation logic.
