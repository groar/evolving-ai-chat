## Manual Scenarios

**Normal flow — blocking check fails → apply_failed**:
When the eval harness returns a result with `blocking: true` and `passed: false`, `_run_evals()` raises `ApplyError("eval_blocked", json.dumps({"blocking_failures": [...]}))`. The apply pipeline catches it and sets `artifact.status = "apply_failed"`, `artifact.failure_reason = "eval_blocked"`. Structured details include `check_type`, `case_id`, `message`, `details` for each failure (sufficient for T-0091 retry).

**Edge case — advisory failure**:
Non-blocking check failures are appended to `artifact.log_text` only; apply continues. Confirmed in code: only when `is_blocking and not expectation_matched` do we append to `blocking_failures` and raise.

**Edge case — run.py timeout**:
On subprocess timeout, _run_evals logs and continues (no raise). Apply can still proceed; timeout is treated as advisory.
