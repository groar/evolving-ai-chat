## Manual Scenario Walkthrough (Code Trace)

### Normal flow: feedback → patch → notification → Undo
1. User submits feedback in FeedbackPanel → "Fix with AI →" button visible per item.
2. Click "Fix with AI →" → `onRequestCodePatch(feedbackId, title, summary)` → `requestPatch("ui")`.
3. `requestPatch` calls `POST /agent/code-patch` → optimistic `pending_apply` entry in store → `setNotificationPatchId(patchId)`.
4. `PatchNotification` renders spinner + "Working on a change based on your feedback…".
5. `schedulePatchPoll(patchId, 1500)` starts. Polls `GET /agent/patch-status/{patchId}` every 1.5 s.
6. Status transitions: `pending_apply` → `applying` → `applied`. Each poll upserts the patch in store.
7. At `applied`: notification shows description + "Undo?" + filled "Undo" button.
8. User clicks "Undo" → `rollbackPatch(patchId)` → optimistic `reverting` → `POST /agent/rollback`.
9. Response `{status: "reverted"}` → store updated → notification shows "Change undone. The app is back to how it was."
10. `refreshState()` re-syncs full state; Changelog shows reverted patch with "See what was reverted ↓".

### Edge case: scope_blocked
1. `POST /agent/code-patch` responds with `{status: "scope_blocked"}`.
2. Optimistic entry set, `schedulePatchPoll` NOT called (terminal on scope_blocked).
3. Notification: "The agent tried to modify files outside the allowed area. Change blocked." + "Dismiss".
4. `refreshState()` called; Changelog shows blocked entry.

### Edge case: apply_failed
1. Poll returns `{status: "apply_failed", failure_reason: "validation_failed"}`.
2. Notification: "Couldn't apply the change: validation_failed. No files were modified." + "Dismiss".
3. Polling stops (terminal state).
