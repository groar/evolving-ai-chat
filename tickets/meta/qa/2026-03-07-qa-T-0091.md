## Manual Scenarios

**Normal flow — retriable failure then retry succeeds**:
_apply_with_retry applies once; on apply_failed with eval_blocked/validation_failed/patch_apply_failed, sets status retrying, saves, builds retry context (reason + details + diff), calls generate_patch with retry_context and existing_artifact_id so same patch_id is returned; updates artifact with new diff/title/description/files_changed; applies again. If second apply succeeds, artifact.status = applied. Verified via unit test test_retriable_failure_triggers_one_retry_and_succeeds.

**Edge case — retry exhaustion**:
If second apply also fails, artifact.failure_reason is set to "retry_exhausted" and user-facing message appended to description. Verified via test_retriable_then_second_apply_fails_sets_retry_exhausted.

**Edge case — non-retriable**:
scope_blocked, base_ref_mismatch, harness_unavailable, patch_timeout do not trigger retry; apply is called once. Verified via test_non_retriable_failure_no_retry.
