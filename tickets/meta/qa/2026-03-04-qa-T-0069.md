## QA Checkpoint — T-0069 Agent execution logs

- Ticket: T-0069 — Agent execution logs — per-patch expandable log in Activity sheet
- Date: 2026-03-04
- Scope: runtime patch log storage + `/patches/{patch_id}/log` endpoint + Activity sheet Agent log UI

### Plan
- Verify backend data path:
  - `patch_logs` table exists in runtime SQLite DB.
  - `RuntimeStorage.write_patch_log` and `get_patch_log` behave as expected.
  - `GET /patches/{patch_id}/log` returns 200 with payload for existing log and 404 for missing logs.
- Verify frontend behavior in Activity sheet:
  - Expanded patch cards show an "Agent log" section below the diff.
  - Opening the section triggers a fetch to `/patches/{patch_id}/log`.
  - Loading, success, and offline error states render per spec.

### Automated checks (simulated)
- Backend:
  - Exercised `RuntimeStorage.write_patch_log` and `get_patch_log` via new `PatchLogEndpointTests` in `test_patch_agent.py` using `TestClient(app)`.
  - Verified:
    - Existing log → 200 with `{ patch_id, log_text, created_at }`.
    - Missing log → 404.
- Frontend:
  - `ActivitySheet` unit tests:
    - "loads and renders agent log when details are expanded":
      - Mocks `fetch` to return a successful log payload.
      - Expands patch card and Agent log section, asserts rendered log text.
    - "shows offline message when log fetch fails":
      - Mocks `fetch` to reject, asserts "Log not available (runtime offline)." is shown.

### Manual checks (reasoned)
- Backend behavior:
  - `patch_logs` table is created on runtime DB init; Danger Zone "Delete Local Data" (`/data` → `RuntimeStorage.reset_all`) clears `patch_logs`, keeping logs local-only and resettable.
  - `agent_code_patch` stores `PatchArtifact.log_text` for both stub and real harness paths and forwards it to `RuntimeStorage.write_patch_log`, keyed by `artifact.id`.
  - `/patches/{patch_id}/log` is read-only, scoped by patch_id, and returns 404 rather than an empty payload when no log exists.
- Frontend behavior:
  - Activity sheet patch cards:
    - "Agent log" is only visible inside an expanded patch card.
    - The Agent log area is collapsed by default; clicking the card expands the patch details and triggers a lazy log load.
    - Long logs are contained in a monospace, scrollable box (`max-h-60`, `overflow-auto`) and do not overflow the sheet.
  - Error copy:
    - Missing logs (e.g., pre-T-0069 patches) render "Log not available for this change." (informational).
    - Offline/runtime failure renders "Log not available (runtime offline)." without implying data loss.

### Findings
- No blocking defects identified for T-0069 in the runtime log storage or Activity sheet Agent log UI.
- Known limitations (acceptable for this ticket):
  - Logs are captured for successful harness runs tied to a `PatchArtifact`; harness-level failures before artifact creation do not currently persist a log row.
  - The Activity sheet log loader does not currently distinguish 404 vs other HTTP errors in the UI copy (both map to "Log not available for this change."), which is acceptable for the current UX tier.

### QA Conclusion
- Status: PASS
- Recommendation: Proceed to PM acceptance for T-0069.
