## Raw Feedback (Sanitized)
"I'd like to have access somewhere (but more as a hidden option or log) to the agent execution logs associated with a change that was made."

## Normalized Summary
When a patch agent runs and produces a change, the user wants to be able to inspect the raw agent execution log (stdout/stderr, tool calls, agent reasoning steps) tied to that specific patch — surfaced as an expandable/hidden detail, not primary UI.

## PM Notes
- "Hidden option or log" signals this is a power-user / debug feature, not a primary flow.
- The natural home is inside the Activity/History panel (T-0067), as an expandable detail section per patch card.
- The runtime currently writes patch logs to the filesystem (patch_agent.py outputs to stdout captured by the harness); a storage/retrieval path for these logs needs to be implemented.
- This has a data-path dependency: the backend must store execution logs keyed by patch ID, and the frontend must retrieve and display them.
- Low-urgency (S3) because it's investigative/debug, not blocking the core loop.

## Triage Decision
- Decision: create T-0069 (Agent execution logs — per-patch expandable log in History panel)
- Rationale: Supports the auditability principle in STATUS.md ("every shipped change ties back to a ticket, diff, and validation evidence"). The log view is the raw evidence.
- Revisit Trigger: after T-0067 (History panel) is accepted; T-0069 depends on T-0067 for its UI home
